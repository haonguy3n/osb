# dm-verity rootfs, anchored in the Secure Boot cmdline

## Why

The signed Unified Kernel Image proves the kernel, initramfs, and command line
are what the project signed. It says nothing about the root filesystem the
kernel then mounts: an attacker who rewrites a block of the rootfs partition
boots a tampered system under a valid signature. dm-verity closes that gap. It
builds a Merkle hash tree over the read-only rootfs and refuses any block whose
hash does not match. The tree's single root hash is the fingerprint of the
entire filesystem, so anchoring that one value extends the signature's trust all
the way into userspace.

The anchor is the **kernel command line**, which already lives inside the signed
UKI. osb computes the root hash at build time and writes it into the cmdline as a
`dm-mod.create=` device-mapper table. The firmware verifies the UKI (cmdline
included) against the enrolled `db`; the kernel then assembles the verity device
from that verified cmdline before mounting root. No initramfs logic, no
bootloader, no separate signature file — the existing Secure Boot signature over
the cmdline is what makes the root hash tamper-evident.

## How it boots

The kernel's `dm-mod.create=` (CONFIG_DM_INIT) creates device-mapper targets
during early boot, straight from the command line, before the root filesystem is
mounted and without any userspace helper. The verity machine's signed cmdline
carries:

```
dm-mod.create="dm-root,,,ro,0 <data_sectors> verity 1 \
  PARTLABEL=rootfs-data PARTLABEL=rootfs-hash \
  4096 4096 <data_blocks> 0 sha256 <root_hash> <salt>"
root=/dev/dm-0 ro rootwait
```

`dm_init` resolves `PARTLABEL=` device references through the kernel's own
`name_to_dev_t`, so no udev, initramfs, or label-resolving userland is needed.
The kernel creates `/dev/dm-0` as the verified view of `rootfs-data`, then
`prepare_namespace` mounts `root=/dev/dm-0` directly (ext4 is built in). A
verity machine therefore ships **no GRUB and no initramfs** — the signed UKI plus
the kernel's built-in device-mapper is the entire boot path. Any modification to
`rootfs-data` makes the block's hash diverge from the tree and the kernel refuses
the read, so a tampered image fails to boot rather than booting compromised.

## Build-time formatting (pure Go, reproducible)

The hash tree is computed in Go on the host during image signing, next to the UKI
build — no `veritysetup` host dependency and no container round-trip. The format
is the standard dm-verity v1 layout: 4096-byte data and hash blocks, SHA-256,
one salt, a superblock followed by the hash tree from the leaves up. The salt is
derived deterministically from the rootfs content (not random), so two builds of
the same rootfs produce the same root hash — reproducibility is preserved.

The result is a `rootfs-hash` partition image and a root hash string. osb writes
the hash image into its GPT partition and substitutes the root hash, salt, and
block count into the machine's cmdline template before signing the UKI.

## Partition layout

A verity machine's disk is:

```
esp           FAT       signed UKI at /EFI/BOOT/BOOT<arch>.EFI
rootfs-data   ext4 (ro) the verified root; never written after build
rootfs-hash   raw       the dm-verity hash tree
```

`rootfs-data` is built exactly like a normal rootfs, then frozen: because verity
is read-only, nothing may write to it after the hash tree is computed. The
`rootfs-hash` partition holds the tree. Both carry stable GPT partition labels so
the cmdline can name them with `PARTLABEL=`.

## Read-only root ergonomics

A verity root cannot be written, so anything the running system expects to write
must live elsewhere. The `rootoverlay` unit handles this: an OpenRC sysinit
service (ordered before bootmisc and every default-runlevel service) mounts one
tmpfs at `/overlay` and layers overlayfs over `/etc`, `/var`, `/root`, `/home`,
and `/srv`, with `/tmp` as a plain tmpfs. The merged view is writable while the
lower (verified) layer stays read-only; writes land in RAM and vanish on reboot,
the intended stateless behaviour for a verified appliance root. Where persistence
is required, a separate writable data partition is the answer. Both verity
machines pull the unit via `distro_packages`, so services that write at boot
(SSH host-key generation, resolv.conf, `/var` state) run unchanged — validated
by booting `ssh-image` on the verity root to an SSH login.

## Kernel configuration

`CONFIG_DM_VERITY`, `CONFIG_DM_INIT`, `CONFIG_BLK_DEV_DM`, and `CONFIG_MD` are
enabled as a `verity.cfg` fragment merged into every kernel, the same way
`container.cfg` and `graphics.cfg` are always merged. dm-verity and dm-init add
little to the image and, like the other fragments, benefit any machine that opts
into them at the cmdline while staying inert for those that do not — which keeps
one shared kernel artifact rather than a verity-specific fork.

## Scope

- **Milestone 1** — the verified-boot chain: kernel config, the Go verity
  formatter, a `verity` machine field, the build/sign path, and a
  `qemu-x86_64-uefi-secureboot-verity` machine that boots `base-image` to a login
  prompt on a `/dev/dm-0` root, validated on x86 then arm64. A deliberately
  corrupted data block must fail the boot.
- **Milestone 2** (done) — the `rootoverlay` writable-overlay unit above, so
  `ssh-image` runs on a verity root.
- **Later** — combining verity with A/B (a hash partition per slot); optional
  `CONFIG_DM_VERITY_VERIFY_ROOTHASH_SIG` for an in-kernel root-hash signature in
  addition to the cmdline anchor.
